# RepoPilot 开发计划

## 第一版 MVP 交付状态

阶段五至八已形成首版代码骨架：结构化补丁与 Maven Recipe 执行、两级 LangGraph 审批、Localhost FastAPI/SSE、React 桌面调试界面和 15 项评测任务定义。真实模型、Embedding、Qdrant 与 Spring Cloud 项目的“研究 -> 计划审批”链路已经联调。

本文件保留第一版分阶段记录，并将后续方向升级为“企业级编程助手平台”。[v2 最小闭环优化设计](docs/RepoPilot-v2-最小闭环优化设计.md) 继续约束任务可靠性；[企业级编程助手平台架构](docs/企业级编程助手平台架构.md) 负责 Skills、MCP、RAG、插件和多语言 Profile 的平台演进。可靠闭环与扩展能力必须并行建设，不能用堆工具代替治理，也不能用治理为由长期停留在单一 Java Demo。

## 1. 开发原则

RepoPilot 不是复刻 Grok Build 的 Rust monorepo，而是借鉴其成熟边界，做一个 45 天内能完整讲清楚、能稳定演示的 Java Coding Agent。

| Grok Build 的启发 | RepoPilot 的取舍 |
|---|---|
| Agent、工具、工作区、会话分层 | 用 Python 模块 + LangGraph 保持分层，不引入大型 Rust 运行时 |
| Plan Mode 先研究后执行 | 将计划审批与执行动作审批分开 |
| 权限规则、Hook、沙箱多层防护 | 首版优先 PolicyGuard、工具白名单、任务权限快照和审计；不夸大 Windows OS 沙箱能力 |
| 工作区与会话可恢复 | SQLite checkpoint + JSONL Evidence + Git 基线/worktree |
| Skills、MCP、插件与子 Agent | Capability Plane 的核心扩展机制；按“注册与信任 -> Transport -> 插件治理 -> 子 Agent”逐层交付 |

核心原则：

1. 一个阶段只交付一个可验证的纵向切片；
2. 先实现拒绝路径，再实现允许路径；
3. 模型只提出工具调用，运行时才决定能否执行；
4. 计划不等于已修复，`PASSED` 必须由证据决定；
5. 所有用户文案、学习文档、README 和代码注释保持中文；
6. 不自动提交、不自动 push、不自动删除用户项目内容。

## 2. 目标架构与领域对象

```text
桌面端 / CLI
    -> API 与事件流
    -> Task Runtime
    -> LangGraph Agent Runtime
    -> Context Broker（RAG / Skill / 上下文预算）
    -> Capability Plane（Built-in / Skills / MCP）
    -> Trust Gateway / Tool Runtime
    -> Workspace Runtime
    -> SQLite / JSONL / Qdrant
```

后续实现围绕以下对象组织，而不是继续把能力堆进 `GraphState`：

| 对象 | 职责 |
|---|---|
| `TaskRequest` | 用户任务、项目引用、任务模式、工作区选择与任务描述 |
| `PermissionSnapshot` | 创建任务时冻结的 safe/full、确认信息、范围与已授权动作 |
| `WorkspaceContext` | 源仓库、基线、实际目录、dirty 状态和 worktree 信息 |
| `ToolDefinition` | 工具名、Pydantic 参数、风险类别、输出上限和执行器 |
| `CapabilityDescriptor` | 内置工具、Skill、MCP 服务/工具的统一来源、作用域、风险和启用状态 |
| `SkillManifest` | 标准 SKILL.md 元数据、内容哈希、渐进加载与工具请求 |
| `McpServerConfig` | Transport、环境变量引用、访问等级、超时、输出上限与工具白名单 |
| `ToolEvent` | 脱敏输入、裁决、摘要、耗时、证据 ID |
| `ChangePlan` | 事实、假设、候选文件、步骤、验证建议与来源 |
| `TaskArtifact` | Diff、测试摘要、报告、交接结果 |

## 4. 阶段五：执行运行时与 Java 验证闭环（第 26-31 天）

### 目标

把“只读计划”升级为“经批准后可执行的代码维护任务”，但仍不开放任意 Shell。

### 实施项

- 引入统一 `ToolRuntime`：工具注册表、Pydantic 输入校验、最大输出、超时、`PolicyGuard` 裁决和标准 `ToolEvent`；
- 完善 `PermissionSnapshot`：创建任务后冻结安全隔离修复或完全本机控制的运行时组合，恢复任务时比对并拒绝权限漂移；
- 增加 `EXECUTION_APPROVAL -> PATCH -> VERIFY -> REVIEW -> REPORT` 节点；
- 实现结构化 `apply_patch`：只允许目标工作区、限制文件/行数、上下文匹配失败即阻断、写后生成 Diff；
- 实现 Maven 专用执行器：仅支持 `compile`、`test`、`-Dtest=<指定测试>` 等 Recipe，使用 argv 而不是拼接 Shell 字符串，记录超时、退出码和 Surefire 摘要；
- 增加 `git_diff`、`create_branch`、worktree 保留/删除/交接前预检；
- 安全隔离修复只写入 Worktree；完全本机控制在 Local 中执行已实现的高风险工具，并持续显示风险与审计；
- 明确 full 的实现边界：本阶段只放行已注册的补丁、Maven 与 Git 交接工具，不提前注册任意 Shell、网络、commit/push。

### 验收

- Java fixture 可在 Worktree 内完成一处小修复并通过 Maven 测试；
- 源仓库的 HEAD、文件哈希和状态保持不变；
- 测试失败报告为 `FAILED`，未运行测试报告为 `UNVERIFIED`，不得输出 `PASSED`；
- 路径逃逸、敏感文件、非法补丁、Maven 参数注入和未批准执行均被拒绝并留痕；
- 任务恢复后可以从同一批准点继续，重复恢复不重复应用同一补丁。

## 5. 阶段六：本地 API 与桌面端（第 32-37 天）

### 目标

交付可演示的本地产品入口，桌面端只负责展示与交互，所有权限判断仍由 Python 后端执行。

### 实施项

- 使用 FastAPI 提供项目、任务、审批、事件、Diff、报告和文档索引 API；
- 用 SSE 推送节点状态、工具进度、审批请求、Maven 输出摘要和任务恢复事件；
- 创建 `Tauri + React + TypeScript + Vite` 工程；
- 实现项目侧边栏、系统文件夹选择器、任务创建页和聊天主界面；
- 实现安全隔离修复/完全本机控制模式卡片、起始 ref、dirty 迁移提示和完全本机控制二次确认；
- 实现计划预览、来源卡片、执行时间线、Diff、测试摘要、审计报告和 worktree 操作页；
- 全量中文文案，full 模式显示持续风险提示。

### 验收

- 用户无需输入路径即可从项目列表创建任务；新项目可通过系统选择器添加；
- 刷新 UI 或重启后端后，任务能根据 `thread_id` 恢复；
- 前端只显示服务端实际返回的权限与工具事件，不能伪造批准状态；
- CLI 保持为同一后端能力的调试入口。

## 9. 每阶段固定验证清单

每次改动至少执行与变更最接近的测试；完成阶段时执行：

```powershell
uv run python -m unittest discover -s tests -t . -v
git diff --check
```

Docker Desktop 可用时追加：

```powershell
docker compose up -d qdrant
uv run repopilot-guard doctor
uv run repopilot-guard bootstrap-qdrant
```

任何依赖缺失、模型调用失败、Qdrant 不可达、审批被拒绝或测试失败都必须如实落为 `BLOCKED`、`FAILED` 或 `UNVERIFIED`。
