# RepoPilot Guard 开发计划

## 1. 开发目标

在 45 天内完成一个专注代码开发的本地 Web Coding Agent。用户通过网页聊天提交 Java/Spring Boot 仓库、Bug/需求和可选研发文档；Agent 结合代码 RAG、文档 RAG、LangGraph 状态图、记忆和受控工具，在 worktree 中提出并验证代码变更，或生成受证据约束的工程文档草稿。

最终 Demo 主线必须是“代码任务闭环”，而不是泛文档问答：

```text
需求/文档 -> 检索代码与资料 -> 分析影响 -> 计划 -> 确认 -> 补丁 -> 测试 -> 证据报告
```

CLI 保留为调试、评测和 CI 入口；Web 是用户体验和面试展示入口。两者共用 LangGraph Workflow 与 `TaskCoordinator`。

## 2. 关键技术决策

| 技术 | 在项目中的职责 | 不承担的职责 |
|---|---|---|
| LangChain | 模型抽象、Embedding、文档加载、切分、Retriever、结构化工具 | 不直接决定工具权限，不替代状态机 |
| LangGraph | Coding Agent 工作流、状态、checkpoint、人工确认、中断恢复 | 不直接执行危险命令，不绕过策略层 |
| Qdrant | 代码/文档向量检索和长期项目记忆，按 Payload 过滤 | 不作为原始文件存储，不保存密钥 |
| FastAPI + SSE | Web API、任务创建、实时事件和报告查询 | 不包含 Agent 业务逻辑 |
| React + TypeScript | 聊天、时间线、上下文来源、diff、测试和报告展示 | 不直接访问本地仓库或 Qdrant |
| Git worktree | 任务隔离与源仓库保护 | 不自动 commit、push 或清理用户分支 |

选择 LangGraph 的原因是它适合有明确节点、人工确认和失败恢复的 Coding Agent：补丁、文档写入和测试都可以在 checkpoint 后暂停，用户确认后从同一状态恢复。选择 Qdrant 的原因是代码/文档 chunk 需要携带项目、提交、路径和页码等 metadata，并按这些字段严格过滤，避免检索到错误项目或旧提交的上下文。

```text

  tools/
    repository.py         文件树、搜索、读取、Java/Maven 分析
    workspace.py          Git worktree 与基线快照
    patch.py              结构化补丁与文档草稿写入
    runner.py             Maven Recipe 受控执行
  policy/
    guard.py              工具、路径、文档和命令策略
  persistence/
    sqlite.py             任务、事件、评测索引

  api/
```

## 4. RAG 与记忆设计

### 4.1 上下文来源

| 来源 | 是否入库 | 关键 Metadata | 说明 |
|---|---|---|---|
| Java 源码、测试、`pom.xml` | 是 | 项目、提交、路径、行范围、语言、哈希 | 只索引通过策略的文件 |
| 仓库内 Markdown/设计文档 | 是 | 项目、提交、路径、标题、哈希 | 与代码同一项目过滤 |
| 用户上传 `.md/.txt/.pdf/.docx` | 是 | 项目、文档 ID、页码/段落、上传时间、哈希 | 仅研发文档，解析失败要可见 |
| 用户聊天内容 | 否，默认不直接入向量库 | 任务 ID | 由 LangGraph checkpoint 保存短期状态 |
| 验证过的任务总结/架构决策 | 是，进入项目记忆库 | 项目、来源任务、验证状态、时间 | 不保存模型猜测和敏感数据 |

### 4.2 索引流程

```text
选择仓库/上传文档
  -> 策略校验与敏感扫描
  -> Loader 解析
  -> 结构化切分
  -> Embedding
  -> Qdrant upsert + Payload
  -> SQLite 记录索引批次与哈希
```

索引的 Payload 必须至少保存：`project_id`、`source_type`、`repo_commit`、`path`、`line_start`、`line_end`、`page`、`document_id`、`content_sha256`。检索永远附带项目过滤；代码检索还附带当前基线提交过滤。

### 4.3 检索策略

1. 先根据任务类型选择检索域：代码修复优先代码，需求驱动改动同时检索需求文档和代码。
2. 使用语义检索召回 Top-K chunk，再按路径、提交、来源类型和文件大小过滤。
3. 只把经过截断和去重的片段放入模型上下文，避免“把整个仓库塞给模型”。
4. 每个片段转为可引用来源，例如 `OrderService.java:84-112`、`订单权限规则.md#3.2`、`需求.pdf p.4`。
5. 检索不足时让模型调用受控的 `search_code`/`read_file`，不能臆造来源。

### 4.4 记忆策略

- 短期：LangGraph checkpoint 保存当前线程消息、计划、已调用工具、待批准补丁和测试结果；
- 工作：Graph State 中只保留经过压缩的“当前假设、候选文件、未解决问题”；
- 长期：仅保存经过测试、用户确认或人工标注的项目事实与任务总结；
- 清理：项目删除或重新索引时，按 `project_id`、`repo_commit` 和文档哈希清理 Qdrant 与 SQLite 索引；
- 禁止：密钥、原始敏感文件、模型臆测、完整私密对话不写入长期记忆。

## 5. 45 天阶段计划

### 阶段 0：安全骨架与产品重新定位（第 1-5 天，已完成）

交付物：PRD、CLI 干跑、显式状态机、Maven Recipe 白名单、敏感路径规则、JSONL 报告和单元测试。

验收：有效 Java Git 仓库只能得到 `UNVERIFIED`；无 Git 或无 `pom.xml` 的仓库得到 `BLOCKED`；没有真实写入和任意 Shell。

### 阶段 1：LangChain/LangGraph/Qdrant 基础设施（第 6-9 天）

目标：让 Agent 能以图状态运行，并为 RAG 打好本地基础。

实施项：

- 在 Python 项目加入 LangChain、LangGraph、Qdrant Client、FastAPI、Pydantic 等依赖；
- 编写 `ModelProvider`、`EmbeddingProvider` 配置接口，所有 Key 从环境变量读取；
- 用 Docker Compose 启动本地 Qdrant，增加健康检查与数据卷；
- 定义 `GraphState`、图节点接口、checkpoint 存储和线程 ID；
- 将现有状态机映射到 LangGraph 节点，但保留 `PolicyGuard` 作为图外强制检查；
- 创建 `coding_context`、`project_memory` Collection 与 Payload 索引脚本。

验收：

- 可启动 Qdrant 并创建 Collection；
- 一个最小 LangGraph 可保存 checkpoint、暂停并恢复；
- 缺失模型 Key/Qdrant 时任务明确 `BLOCKED`，不降级成无证据成功；
- 现有 8 个控制平面测试仍通过。

### 阶段 2：Git worktree 与受控代码工具（第 10-13 天）

目标：完成真实代码操作前的隔离层。

实施项：

- 实现 `WorkspaceManager`、`GitClient` 和基线快照；
- 创建 detached worktree，记录源仓库提交、dirty 状态和初始 diff；
- 实现 `list_files`、`search_code`、`read_file`、`inspect_build` 工具；
- 所有工具统一经过路径、敏感文件、大小、深度和结果数量限制；
- 源仓库 dirty 时默认阻塞，绝不 stash/commit；
- 建立临时 Java Git fixture 测试。

验收：源仓库文件哈希和 Git 状态不变；worktree 内读取正确；路径逃逸、`.env`、`.git`、生产配置均被拒绝并留痕。

### 阶段 3：代码与研发文档 RAG（第 14-18 天）

目标：让 Agent 能可靠拿到“任务相关且可引用”的代码和文档上下文。

实施项：

- 实现仓库代码、`pom.xml`、Markdown 文档的 Loader 与结构化切分；
- 实现上传 `.md/.txt/.pdf/.docx` 的校验、解析、存储和失败反馈；
- 实现 Embedding、Qdrant upsert、按哈希增量更新和项目/提交过滤；
- 实现 `retrieve_context` 工具，返回内容、来源、分数和 metadata；
- 为检索结果加上下文预算、去重、截断与来源格式化；
- 对上传内容执行提示注入提示与敏感模式扫描，内容只作为数据进入 RAG。

验收：

- 同一问题可检索到当前仓库指定提交中的代码 chunk；
- 文档片段能显示文件名、标题或页码；
- 不同项目/旧提交/敏感文件的 chunk 不会被召回；
- 为至少 3 个代码任务建立“代码 + 文档”的检索单测与人工样例。

### 阶段 4：LangGraph Coding Workflow 与记忆（第 19-24 天）

目标：让 Agent 以明确、可暂停的 Coding 流程工作。

实施项：

- 实现 `intake -> preflight -> ingest -> retrieve -> analyze -> plan` 节点；
- 使用 LangChain Structured Tool 注册检索与仓库工具，统一记录工具输入输出；
- 计划阶段只开放只读工具，输出根因假设、候选文件、改动计划和验证计划；
- 在补丁、文档写入和高成本测试前使用 LangGraph interrupt 等待确认；
- 实现 checkpoint 恢复、会话摘要和项目长期记忆写入规则；
- 将真实测试失败摘要写入工作记忆，限制重复调用与循环次数。

验收：浏览器刷新或服务重启后，任务能从待确认步骤继续；模型无法在计划阶段调用写入工具；长期记忆仅保存已验证事实。

### 阶段 5：补丁、Maven 与工程文档草稿（第 25-30 天）

目标：完成真实但受控的代码变更闭环，并支持“代码上下文驱动的文档生成”。

实施项：

- 实现结构化 patch 解析、上下文匹配、文件/行数限制和统一 diff；
- 实现 `SafeCommandRunner`，仅用 argv 执行 Maven `compile/test/targeted_test`；
- 解析测试退出码、Surefire 失败摘要、耗时和超时；
- 实现 `write_document_draft`：生成 PRD、技术方案、接口说明、测试计划或变更说明草稿；
- 所有 patch 和文档写入先进入 `pending_approval`，确认后才写入 worktree；
- 将测试结果、diff 和来源引用汇入报告。

验收：人工确认后可在 worktree 完成一处代码修改或一份文档草稿写入；测试失败绝不返回 `PASSED`；源仓库不变。

### 阶段 6：FastAPI、SQLite、SSE 与 Web Chat（第 31-35 天）

目标：把真实 Agent 工作流呈现为可用的网页产品。

实施项：

- 建立 FastAPI 任务、上传、事件、审批、diff、报告、评测 API；
- SQLite 保存任务索引、事件序号、产物索引、索引批次和评测结果；
- 使用 SSE 推送 LangGraph 节点、工具、审批和测试事件，支持断线回放；
- 初始化 React + TypeScript + Vite；
- 完成聊天输入、上传区、任务详情、来源卡片、执行时间线、diff、测试和报告页签；
- UI 全部使用中文，状态颜色必须区分成功、失败、阻止和未验证。

验收：浏览器能上传文档、创建任务、查看来源、确认补丁、实时查看测试与报告；CLI 和 Web 调用同一图工作流。

### 阶段 7：评测、RAG 对照与安全回归（第 36-41 天）

目标：得到可解释的评测结果和安全证据。

实施项：

- 建立 15 个固定 Java/Spring Boot 维护任务；
- 其中至少 5 个提供需求/API/设计文档，分别运行“代码检索”与“代码 + 文档 RAG”；
- 记录任务成功、目标测试、定位命中、引用正确性、Token、嵌入成本和安全拦截；
- 新增文档提示注入、跨项目检索、旧提交检索、敏感索引、路径逃逸、命令注入回归测试；
- 生成评测 CSV/JSON 与 Web 汇总表。

验收：任务和文档版本可固定复现；失败记录完整保留；能解释文档 RAG 对成功或失败的影响。

### 阶段 8：Demo、文档与简历打磨（第 42-45 天）

目标：将能力转化为可被面试官快速理解的项目作品。

实施项：

- 完成中文 README、架构图、环境变量示例、Docker Compose 启动说明和已知限制；
- 准备四个 Demo：成功修复、文档辅助修复、测试失败、策略/提示注入拦截；
- 录制 3 至 5 分钟 Web Demo；
- 整理评测表、失败案例、简历表述和面试讲解脚本；
- 运行后端、前端、RAG、API 和安全回归检查。

验收：新环境可按 README 启动；每个 Demo 都能展示来源、状态、diff、测试和最终结论；公开文档、UI 和代码注释均为中文。

## 6. 每日开发规则

1. 每天先完成一个可验证垂直切片，不同时打开多个未完成模块。
2. 改动 Agent 流程时优先补 LangGraph 节点/状态转移测试。
3. 改动检索时优先验证 metadata 过滤和来源引用，再看回答质量。
4. 改动工具时先验证策略拒绝路径，再验证正常路径。
5. 每个功能至少有一次真实 Java Git fixture 冒烟；涉及写入时必须对比源仓库与 worktree。
6. 记录模型、Prompt、Embedding、任务版本和预算，评测时禁止临时替换。

## 7. 风险与控制

| 风险 | 控制措施 |
|---|---|
| LangChain/LangGraph 代码量膨胀 | 只实现一个 Coding Graph，节点职责固定，不引入多 Agent |
| 向量检索变成“全库乱召回” | 强制 `project_id + repo_commit + source_type` 过滤，保留来源卡片 |
| 上传文档提示注入 | 文档仅作为不可信数据，不能修改工具策略；界面标注可疑文本 |
| 45 天内前后端压力大 | 页面只做聊天、详情和评测表，不做登录、主题系统、复杂图表 |
| 模型成本和效果不稳定 | 限制图循环/检索预算，固定模型和任务集，完整保留失败证据 |
| Maven 环境不稳定 | 固定 Java/Maven 版本，记录基线，优先 Maven Wrapper，必要时 Docker 化 |
| 误改用户仓库 | worktree 强制隔离，dirty 源仓库阻塞，禁止 commit/push/reset/clean |

## 8. 最终完成定义

- Web 端可完成代码任务、文档上传、来源检索、人工确认、补丁、测试和报告闭环；
- LangGraph checkpoint 能让任务暂停与恢复，LangChain 工具受 `PolicyGuard` 约束；
- Qdrant 中的代码、文档和长期记忆有来源 metadata，且检索不会跨项目或跨提交泄漏；
- 文档生成仅产生代码相关工程文档草稿，写入前可审查；
- 15 个代码维护任务和至少 5 个文档辅助对照任务有可重放评测数据；
- 成功、失败、阻止、提示注入拦截四类演示可稳定复现；
- 所有面向用户的文档、UI、报告和代码注释为中文。
