# 插件包规范与学习实验

## 1. 为什么 Coding Agent 需要插件治理

Skill、MCP 配置和项目规则本质上都是外部输入。它们可以帮助 Agent 了解团队规范，也可能被人误改、被同步程序覆盖，或包含提示注入。因此 RepoPilot 不采用“目录里有文件就自动执行”的方式，而是把插件当作需要登记的本地软件包。

插件的作用是复用知识和配置，不是绕过安全边界。它不能提高任务权限，不能改变 `PolicyGuard`，也不能让模型获得任意 Shell。

## 2. 一个最小插件

```text
spring-maintenance/
  repopilot-plugin.json
  skills/
    spring-maintenance/
      SKILL.md
```

`repopilot-plugin.json` 的最小结构如下：

```json
{
  "schema_version": 1,
  "id": "spring-maintenance",
  "name": "Spring Boot 维护规范",
  "version": "0.1.0",
  "description": "为 Spring Boot 仓库提供维护检查清单。",
  "skills_root": "skills"
}
```

可选的 `mcp_config` 只能是插件根目录内的相对路径。当前版本只展示和审计该配置，绝不自动连接其中的服务。

## 3. 安装与完整性

执行 `plugin install` 时，RepoPilot 会校验清单字段和所有声明路径，并对整个包内的普通文件按“相对路径 + 文件 SHA-256”计算确定性哈希。符号链接、路径逃逸、单文件超过 1 MiB、文件过多或无效 JSON 都会被阻断。

```powershell
uv run repopilot-guard plugin install --source .\examples\plugins\spring-maintenance
uv run repopilot-guard plugin list
```

之后每次读取活跃插件时都会重新计算哈希。如果有人修改了 `SKILL.md`，状态会变为 `TAMPERED`，插件不会再提供 Skill 根目录。用户可以先审查 diff，再执行同一条 `plugin install` 作为显式重新安装。

## 4. 启用状态与审计

插件默认登记为启用，但仍须完整性通过才会生效。`disable` 只修改 SQLite 中的启用状态，不删除源目录；`remove` 只解除登记，也不删除用户文件。所有操作都可使用 `plugin audit` 查看。

```powershell
uv run repopilot-guard plugin disable --plugin-id spring-maintenance
uv run repopilot-guard plugin enable --plugin-id spring-maintenance
uv run repopilot-guard plugin audit --plugin-id spring-maintenance
```

当前落地的是安全控制面。下一步才是任务级插件选择、MCP 配置合并与 Hook 协议；每一步都必须复用既有的 Capability Policy、审批和 Evidence 机制。
