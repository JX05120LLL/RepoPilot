# RepoPilot 企业级编程助手平台架构

> 文档状态：v3 架构基线
>
> 核心结论：RepoPilot 不再只是一个 Java 修复工作流，而是“本地编程助手平台 + Java/Maven 首个深度 Profile”。Skills、MCP、RAG 和内置工具共享同一能力、权限与审计体系。

## 1. 参考与取舍

本设计参考 Codex 与 Grok Build 的公开架构，不复制其源码：

- Codex Skills 使用 `SKILL.md` 封装指令和资源，发现阶段只提供名称、描述和路径，选中后才加载正文，避免所有 Skill 永久占用上下文；
- Codex MCP 支持本地 STDIO 和远程 Streamable HTTP，并在 CLI、IDE、桌面入口共享配置；
- Grok Build 将 TUI、Agent Shell、Tools、Workspace、Config/MCP/Sandbox 拆为独立 crate，说明界面、推理、能力和执行不应混在一个 Agent 类里；
- Grok Build 的 MCP 运行时具有连接状态、超时、重启和工具命名空间，Skills 具有多作用域发现、优先级和 `allowed-tools`。

RepoPilot 的取舍：继续使用 Python、LangGraph、FastAPI 和 Tauri，不迁移到 Rust；先建立稳定接口和治理边界，再逐步增加真实 Transport 与插件生态。

参考资料：

- [Codex Skills](https://developers.openai.com/codex/skills)
- [Codex MCP](https://developers.openai.com/codex/mcp)
- [Grok Build](https://github.com/xai-org/grok-build)

## 2. 目标架构

```text
Tauri Desktop / CLI / IDE Adapter
                 |
        Local API + SSE/WebSocket
                 |
Task Service --------------------------------------+
任务、会话、审批、租约、取消、Artifact、事件       |
                 |                                 |
LangGraph Orchestrator                             |
研究、计划、执行、验证、审阅、恢复                 |
                 |                                 |
Context Broker             Capability Plane        |
RAG / 摘要 / 项目规则      Built-in Tools          |
Skill Catalog / Token预算   Skills / MCP / Plugins  |
                 |                 |               |
                 +-------- Trust Gateway ----------+
                          权限快照
                          PolicyGuard
                          审批 / 参数 Schema
                          超时 / 输出上限 / 审计
                                 |
                         Execution Plane
                 Workspace / Git / Patch / Maven
                 MCP Transport / Skill Script Runner
                                 |
                    SQLite / Qdrant / JSONL / Files
```

## 3. 六个核心平面

### 3.1 Task Plane

任务是权限、工作区和证据的最小边界。每个任务冻结 `task_id`、`thread_id`、项目、基线、任务模式、权限快照、能力快照和验证契约。恢复时必须验证这些快照一致，不能因为服务重启而升级权限。

### 3.2 Agent Plane

LangGraph 只负责编排，不拥有文件系统、进程或网络权限。模型提出结构化意图；节点决定何时可以研究、何时暂停审批、何时调用执行器。未知节点跳转、未知工具和不符合 Schema 的参数统一拒绝。

### 3.3 Context Plane

Context Broker 统一管理五类上下文：

| 类型 | 来源 | 生命周期 |
|---|---|---|
| 当前任务 | 用户消息、计划、工具结果、Diff | SQLite checkpoint，会话级 |
| 代码上下文 | 源文件、构建文件、测试 | Qdrant `coding_context`，项目+提交级 |
| 研发文档 | MD/TXT，后续 PDF/DOCX | Qdrant，项目/文档级 |
| 项目记忆 | 经过验证的架构事实和决策 | Qdrant `project_memory`，可撤销 |
| Skill 指令 | 被选中的 `SKILL.md` | 按需加载，任务级 |

“百万上下文”是模型一次请求可处理的 Token 上限，不是让应用把整个仓库、所有 Skill 和全部历史一次塞进去。企业产品仍需要预算、检索、去重、压缩和来源追踪。

### 3.4 Capability Plane

所有可被 Agent 使用的能力都登记为 `CapabilityDescriptor`：

```text
capability_id / kind / scope / source
description / enabled / risks / metadata
```

能力类型包括内置工具、Skill、MCP Server、MCP Tool，后续增加 Hook 和子 Agent。作用域优先级为 `project > user > bundled`。风险可以组合：`read/write/process/network/secret_access`。

### 3.5 Trust Plane

信任判断顺序固定：

```text
组织硬拒绝
  -> 任务权限快照
  -> 能力启用和来源信任
  -> 当前动作审批
  -> 参数/JSON Schema
  -> 路径、工作区、Recipe
  -> 执行器
  -> 输出限长、脱敏、Evidence
```

Skill、MCP server instructions、代码注释、README、Issue 和 RAG 片段都是不可信数据。它们可以建议下一步，但不能修改系统提示、权限快照、能力目录或图路由。

### 3.6 Execution Plane

执行器按能力类型隔离：

- 文件与 Git：工作区边界、结构化补丁、基线校验、原子写入；
- 构建与测试：语言 Profile 提供固定 argv Recipe，不拼接 Shell；
- MCP：Transport 负责连接，Capability/Policy 决定工具是否可见和可调用；
- Skill Scripts：未来进入受控脚本运行器，不因位于 Skill 目录就自动可信；
- Artifact：大输出溢出到任务目录，事件只保存摘要、大小和哈希；计划、补丁、Diff、验证和报告在内容变化时保留不可变版本，当前版本仅是方便审阅的指针。

## 4. Skills 设计

### 4.1 文件约定

```text
<skill-name>/
  SKILL.md              # 必需
  scripts/              # 可选，默认不执行
  references/           # 可选，按需读取
  assets/               # 可选，不进入默认上下文
```

项目级路径支持 `.agents/skills` 和 `.repopilot/skills`。`SKILL.md` 使用 YAML frontmatter，至少包含 `name` 和 `description`，可选 `allowed-tools`、`user-invocable`、`disable-model-invocation`、`compatibility`。

### 4.2 渐进披露

1. Discover：只读取、校验并登记元数据与 SHA-256；
2. Catalog：按固定字符预算提供名称、描述、路径；
3. Select：用户显式选择或模型根据描述匹配；
4. Load：重新校验内容哈希后加载正文；
5. Authorize：`allowed-tools` 与任务能力集合求交集；
6. Execute：每个工具仍独立经过 Trust Gateway。

当前实现完成 1-4 和能力登记，并已由 Context Broker 完成项目 Skill 的确定性匹配、哈希复验、字符预算与任务快照。`allowed-tools` 依然只是请求，不能改变已绑定工具；Skill 脚本尚未注册执行器。

## 5. MCP 设计

### 5.1 配置原则

```toml
version = 1

[[servers]]
name = "docs"
transport = "streamable_http"
url = "https://mcp.example.com/v1"
scope = "project"
access = "read_only"
bearer_token_env = "DOCS_MCP_TOKEN"
tool_timeout_seconds = 30
max_result_chars = 20000
```

- 禁止 `api_key/token/password/headers` 等内联密钥；
- HTTP 除 localhost 外必须 HTTPS，URL 禁止 userinfo、query 和 fragment；
- STDIO 使用 command + args，不通过 Shell 拼接；裸 `python`/`python3`/`python.exe`/`python3.exe` 固定解析为启动 RepoPilot 的解释器，避免 Windows `PATH` 漂移；带目录的相对路径、绝对路径和其它命令保持原样；
- 工具统一命名为 `mcp__server__tool`；
- 配置成功只表示 `CONFIGURED_NOT_CONNECTED`，握手成功后才能是 `READY`。

### 5.2 连接状态

```text
DISABLED -> INITIALIZING -> READY
                    |-> NEEDS_AUTH
                    |-> UNAVAILABLE -> RETRYING -> READY
```

当前运行时已基于官方 MCP Python SDK `>=1.28.1,<2` 实现 STDIO/Streamable HTTP、超时、有限重试、熔断、并发连接合并、单 Server 串行调用与确定性关闭。工具发现会重建当前 Server 的 Capability 快照；CLI/API 探测使用显式短连接，避免后台悄悄持有高权限会话。

已接入 LangGraph 的任务级 MCP Binding：安全模式默认不连接，用户只能批准明确的 `mcp__server__tool`；完全本机控制也只绑定项目配置标为 `read_only` 的工具。`MCP_BINDINGS` 节点短连接发现工具，冻结配置与 Schema 哈希，并把已绑定工具写入 Context Broker/SQLite checkpoint。每次调用会再次校验配置哈希和 Schema，变化即 `BLOCKED`；Server instructions 和工具输出始终是不可信数据。

仍需补齐任务生命周期持久连接池、服务级并发配额、OAuth 和手动重连 UI。运行中的任务不会自动获得新发现工具，也不会因项目目录出现 `mcp.toml` 而自动连接。

## 6. RAG 与长期记忆

Qdrant 负责“按语义找相关内容”，不保存 LangGraph 的执行位置；SQLite 负责任务、checkpoint、项目注册、能力配置索引和 Artifact 元数据。二者职责不能互换。

当前 Context Broker 会把项目 `AGENTS.md`、项目规则文件、匹配的 Skill、当前提交 RAG 片段、同项目已验证记忆和任务实际绑定的只读工具组装为不可信上下文包；只保存来源、哈希、预算和能力快照到 SQLite checkpoint，避免把完整代码或 Skill 正文混入任务状态。MCP 只有经过任务授权、发现、冻结和复验后才会进入模型工具清单。

`project_memory` 已实现写入边界：只有 `REVIEW` 节点同时取得真实 Diff 与 Maven `PASSED` 后，才按真实修改文件写入一条确定性事实；它包含基线提交、任务 ID、Diff/验证哈希和 Maven 结果，不包含模型计划、用户描述或任何推测。检索仅过滤同一 `project_id + verified=true`，因此可跨提交帮助后续维护任务，但每条来源仍标明原基线提交。记忆服务失败只产生审计事件，绝不把已验证的 `PASSED` 改写为失败。

下一步 RAG 演进：

1. 已完成 Vector 候选集内的关键词/路径确定性重排，结果保留混合分、向量分与词法分；
2. 增加独立 BM25 与 Java AST/符号关系、调用链辅助召回；
3. Cross-encoder 或模型重排；
4. 按 Token 预算拼装上下文；
5. 每个片段保留项目、提交、路径、行号、哈希和可信状态；
6. 已实现：只有测试通过后的确定性变更事实进入 `project_memory`；后续增加人工确认、撤销和保留策略。

## 7. 企业级非功能要求

| 维度 | 最低要求 |
|---|---|
| 安全 | 默认最小权限、密钥不落盘、路径/Schema/命令校验、供应链来源可追踪 |
| 可靠性 | 幂等任务、租约、心跳、可恢复 checkpoint、外部调用超时/重试/熔断 |
| 可观测性 | 结构化事件、Trace ID、节点/工具耗时、Token/成本、错误分类和脱敏日志 |
| 可评测 | 固定 fixture、修复前失败、修复后验证、范围断言、安全断言、可重放报告 |
| 可扩展 | 稳定 Capability/Skill/MCP/Profile 接口，Schema 版本和迁移策略 |
| 可运维 | 配置 doctor、连接状态、健康检查、版本兼容、数据备份/清理策略 |

当前已经实现最小可关联 Trace：服务端为每个任务生成不可由客户端指定的 `trace_id`，它随 SQLite 任务记录持久化，并出现在该任务全部 SSE 事件和桌面任务详情中。旧数据库会在首次打开时回填一次。节点级耗时、模型 Token/成本和 OpenTelemetry 导出仍是后续工作，不能把当前 Trace ID 误称为完整可观测平台。

## 8. 最近三轮迭代顺序

1. **P1 已完成**：Capability Registry/Policy、Skill Registry、MCP Config Registry、CLI 与测试；
2. **P2 核心已完成**：真实 STDIO/Streamable HTTP、连接状态机、工具发现/调用网关、CLI/API/桌面显式探测，以及任务级 MCP 动态工具绑定；后续补 OAuth、持久池、并发配额与 Artifact 溢出；
3. **P3 首个闭环已完成**：Context Broker 已将项目规则、RAG、Skill 和任务冻结的内置/MCP 只读工具按预算组合并冻结快照；下一步补会话摘要、BM25/符号索引和模型重排。

本轮新增 Artifact Versioning：SQLite 保存版本、哈希、大小和时间，正文写入任务目录 `history/`；同一内容的 checkpoint 重放不制造新版本，读取时重新校验 SHA-256。API 和桌面端只能查询并读取指定版本，不能覆盖或删除历史。

每轮都必须同时交付代码、测试、中文文档、失败路径和可演示证据，不以“模型回答看起来正确”作为验收。
