# 阶段二学习实验：隔离工作区与权限模式

本实验帮助你理解 RepoPilot Guard 为什么在真正修改代码前，先创建 Git worktree 和权限边界。

## 1. 先理解三个目录

```text
源仓库
  D:\code\sample-spring-app

任务产物目录
  %TEMP%\repopilot-guard\runs\task-xxxx

隔离 worktree
  %TEMP%\repopilot-guard\runs\task-xxxx\worktree
```

源仓库是用户正在开发的代码。任务 worktree 是 Git 根据某个提交创建的独立副本。后续 Agent 修改代码时，默认只修改 worktree；因此即使补丁有问题，源仓库文件也不会被改动。

## 2. Git worktree 和 clone 的区别

`git clone` 会复制整个仓库和 `.git` 数据。`git worktree` 会复用同一个 Git 对象库，为指定提交创建额外工作目录，创建速度更快、磁盘占用更小。

RepoPilot Guard 使用 detached worktree：它固定在创建时的 commit，不绑定用户正在使用的分支。这样后续 Agent 的改动不会移动用户分支，也不会自动 commit。

## 3. 运行安全模式实验

准备一个已经提交、没有未提交改动的 Maven 仓库：

```powershell
uv run repopilot-guard workspace prepare --repo D:\code\sample-spring-app --task "学习 worktree"
```

输出中的关键字段：

```text
snapshot.head_commit       创建时的 Git 提交
snapshot.is_dirty          源仓库是否有未提交改动
workspace_path             隔离副本目录
source_unchanged           创建前后源仓库内容是否一致
evidence_events_path       本次任务审计事件
```

试着在源仓库创建一个未提交文件，再运行同一命令。安全模式会返回 `BLOCKED`。系统不会替你 `stash`、`commit`、`reset` 或删除文件。

## 4. 完全权限模式

完全权限是按任务授予的，必须输入固定确认语句：

```powershell
uv run repopilot-guard workspace prepare --repo D:\code\sample-spring-app --task "学习完全权限" --permission full --confirm-full-access "我已了解完全权限风险"
```

完全权限模式会记录 `USER_GRANTED_FULL_ACCESS`。当前阶段它允许 dirty 源仓库继续创建 worktree，但 worktree 仍只基于最新提交 `HEAD`，不会携带未提交改动。

这很重要：完全权限是一种未来工具的授权能力，不代表当前阶段已经实现任意 Shell、联网、补丁、删除、commit 或 push。

## 5. 只读工具的安全规则

阶段二实现了四个只读工具：

- `list_files`：最大深度 6、最多 200 项。
- `search_code`：字面量搜索、最多 100 个匹配。
- `read_file`：只读 UTF-8 文本，单文件不超过 256 KiB。
- `inspect_build`：读取 `pom.xml` 和 Maven Wrapper 信息，不执行 Maven。

安全模式下，`.env`、`.git`、`.ssh`、证书文件、生产配置和 worktree 外路径都会被 `PolicyGuard` 拒绝。完全权限模式会放行并记录风险授权；后续真正高风险工具开放前，仍会在界面上要求用户确认。
